# Beacon V1 Contract

This file records the Milestone 0 decisions that define the v1 release gate.
Tests in later milestones should implement this contract instead of expanding
scope ad hoc.

## Release Gate Commands

Repository validation commands:

- `git submodule update --init --recursive`
- `just fmt`
- `just lint`
- `just test`
- `mdbook build docs`
- `pnpm --filter beacon-lsp compile`
- `pnpm --filter beacon-lsp lint`

`pnpm install` is setup, not a gate command, but editor package checks assume
dependencies are installed.

Product commands in the v1 contract:

- `beacon typecheck PATHS`
- `beacon typecheck --format json PATHS`
- `beacon typecheck --format compact PATHS`
- `beacon analyze file PATH`
- `beacon analyze package PATH`
- `beacon analyze project PATH`
- `beacon lint PATHS`
- `beacon format --check PATHS`
- `beacon format --write PATHS`
- `beacon lsp --stdio`
- `beacon lsp --tcp --host HOST --port PORT`
- `beacon version`

`parse`, `check`, `resolve`, `highlight`, function/class analysis targets, and
debug subcommands are developer tools. They can be tested, but they are outside
the v1 release gate unless a later milestone promotes them.

Commands that print placeholder output are outside the v1 path until they are
implemented or hidden.

## Fixture Contract

The v1 supported Python subset is defined by `fixtures/workspace/` and focused
crate fixtures that reference the same behavior.

`fixtures/workspace/` is the cross-frontend contract fixture. Parser,
analyzer, CLI, and LSP tests should load it through shared helpers instead of
copying files.

Focused fixtures are allowed when they test crate-local behavior that would
make the shared workspace noisy. They should not define a different product
contract.

## Diagnostic Policy

V1-stable diagnostic fields:

- code
- severity
- primary span
- source file
- optional tags
- machine-readable JSON field names

Full message text may improve over time. Tests should assert stable message
fragments rather than entire prose unless the exact text is part of a UI
contract.

CLI and LSP diagnostics for the same workspace must agree on code, severity,
span, and suppression behavior. Output format differences are presentation
only.

`beacon typecheck --format json` is a v1 contract surface. The JSON payload is:

```json
{
  "schema_version": 1,
  "diagnostics": [
    {
      "file": "path/to/file.py",
      "code": "HM001",
      "severity": "error",
      "message": "stable enough for humans, not exact-test-only text",
      "span": {
        "start": { "line": 1, "col": 1 },
        "end": { "line": 1, "col": 5 }
      },
      "tags": []
    }
  ],
  "failures": [
    { "file": "path/to/file.py", "error": "read failure" }
  ]
}
```

Line and column values are one-based in CLI JSON. Diagnostic tags are lowercase
LSP tag names such as `unnecessary` and `deprecated`; diagnostics without tags
use an empty array. The golden fixture for this shape lives with the CLI tests.

After v1, removing a diagnostic code, changing a code's meaning, changing JSON
field names, or changing default severity requires an explicit compatibility
note and migration path.

## Config Policy

Supported config sources:

- `beacon.toml`
- `[tool.beacon]` in `pyproject.toml`
- LSP/editor initialization settings
- file mode directives such as `# beacon: mode=strict`

Precedence, from lowest to highest:

- defaults
- `pyproject.toml`
- `beacon.toml`
- LSP/editor settings
- file mode directives for the narrow options they support

Invalid config must be reported. CLI runs should fail with a non-zero status
for invalid config. LSP runs should publish or surface a configuration
diagnostic instead of silently ignoring the setting.

After v1, removing a config key, changing its type, or changing default
behavior requires an explicit compatibility note and migration path. Adding
optional keys is allowed when defaults preserve existing behavior.
